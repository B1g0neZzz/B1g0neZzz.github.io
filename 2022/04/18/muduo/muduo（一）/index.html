发表于 2022-04-18 更新于 2022-04-24 分类于 muduo

一个典型的服务器结构

  • 网络 I/O + 服务器高性能编程技术 + 数据库
  • 超出数据库连接数
    • 数据库并发连接数 10 个，应用服务器这边有 1000 个并发请求，将会有 990 个连接失败
      • 解决方法：队列 + 连接池，
  • 超时时限
    • 数据库并发连接数 10 个，数据库 1 秒内最多能处理 1000 个请求，应用服务器这边有 10000 个并发请求，会出现 0-10 秒的等待。
      • 解决办法：
        1. 主要业务处理挪到应用服务器处理，数据库只做辅助的业务处理，缓解数据库的压力。
        2. 缓存

使用缓存有几个问题：

缓存更新（缓存同步），缓存 time out

如果缓存失效，重新去数据库查询，实时性比较差。

一旦数据库中数据更新，立即通知前端的缓存更新，实时性比较高。

缓存换页问题： 内存不够，将不活跃的数据换出内存。FIFO LRU LFU（最不频繁使用）

如果使用了缓存，数据库依然不能满足并发量怎么办？

将数据库实行读写分离，主数据库 master 实现 write 操作，从数据库 slave 实现 read。一旦主数据库数据改变，就需要同步数据库，这就需要 Replication 机制。这就实现了服务器的 负载均衡。

对应用服务器的负载均衡：

第一种方案：增加一个任务服务器来实现，任务服务器可以监视应用服务器的负载， CPU 高、 I/O 高、并发高、内存换页高。查询到这些信息之后，选取负载最低的服务器分配任务。

第二种方案：应用服务器主动到任务服务器中主动取，这是更加 公平 的。

举例，一个服务器有 3 个任务，另一个服务器有 5 个任务。但是有 5 个任务的服务器因为处理的任务比较简单，处理起来更快，但是任务服务器可能会认为 3 个任务的服务器负载更小，这样就造成了不公平。

但是第二种方案也有缺点，就是如果应用服务器只能处理一些 特定 的任务，就会增加 任务服务器 的复杂度。但是如果所有的应用服务器都处理 相同 的任务，第二种方案就更加科学。

同时，任务服务器也应该有多台，一台任务服务器挂掉，另一台任务服务器马上接应。

知识点1：signal(SIGPIPE)

为什么要 signal(SIGPIPE) 呢？如果客户端关闭套接字 close ，而服务器调用了一次 write，服务器会接收一个 RST Segement，如果服务器端再次调用了 write，这个时候就会产生 SIGPIPE 信号，SIGPIPE 信号会导致进程终止。

RST Segement： TCP 协议中重置、复位连接的标志位，用来关闭异常连接。

发送 RST 包关闭连接时，不等缓冲区的包发送完成，会丢弃缓冲区中的包，直接发送 RST。同样接收端收到 RST 包后，也不必发送 ACK 确认包。

RST Segement 使用场景:

  1. 当服务器收到没有接听端口的连接请求时会返回 RST 包
  2. 异常终止一个连接
  3. 检测半打开连接（ 服务器突然掉电，客户端不知道，依然向服务器端发送请求，服务器来电后向客户端发送 RST ）

知识点2： TIME_WAIT

应尽可能在服务器端避免出现 TIME_WAIT 状态。如果服务器端主动断开连接（先于 client 调用 close ），服务端就会进入 TIME_WAIT。

协议设计上，应该让客户端主动断开连接，这样就把 TIME_WAIT 状态分散到大量的客户端。

如果客户端不活跃了，一些客户端不断开连接，这样子就会占用服务器端的链接资源。服务器端也要有个机制来踢掉不活跃的连接（使用 close），虽然这样会让服务器端进入 TIME_WAIT 状态，但这毕竟是少量情况。

知识点3：关于 --it;

例如现在 it 指向 1 这个元素，erase 后 it 的状态时不确定的，可能会变成野指针，所以要把 erase 后的返回值重新赋值给 it。

现在 it 指向 6，循环中会绕过 6 这个元素，所以要 –it。

如果发送缓冲区只有 1000，而要发送的数据有 10000，该怎么办？

POLLOUT 事件触发时机：当发送缓冲区不再满时触发。

为什么不在一开始就关注 connfd 的 POLLOUT 事件呢？因为一开始发送缓冲区都是空的，每次都会触发事件，会出现 busy-loop。

EMFILE 处理

accept 返回 EMFILE 意味着进程打开了太多的文件。

处理方式：

  • 调高进程文件描述符的数目
  • 死等
  • 准备一个空闲的文件描述符。 遇到这种情况，先关闭这个空闲文件，获得一个文件描述符名额；再 accept 拿到 socket 连接的文件描述符；随后立刻 close，这样就优雅地断开了与客户端的连接；最后重新打开空闲文件，把 “坑” 填上，以备再次出现这种情况时使用。

ET 模式

为什么 read 直到返回 EAGAIN 错误？

因为是 ET 模式，第一次数据来时，数据是从无到有的状态，我们可以认为是低电平到高电平的变化使得 ET 模式的 epoll 感应到数据到来。假设数据有 1000，而我们只读了 100，数据还有 900，后续再有数据来的时候，依然处在高电平状态，不会再触发 EPOLLIN 事件了。

write 到 EAGIN 是同理的。

同时，ET 模式有个问题。例如 accept 事件没有成功接收连接（比如因为 EMFILE ），一直处于高电平，后续的连接都不会再触发 EPOLLIN 事件。

事件回调

基于虚函数的线程类

在传统的 C++ 程序中，事件回调是通过虚函数进行的。网络库往往会定义一个或几个抽象基类（Handler class），其中声明了一些虚函数，如 onConnect()、 onMessage() 等。使用者需要继承这些基类，并覆盖（override）这些虚函数，以获得事件回调通知。

然后使用基类的指针指向派生类的对象，这样就可以通过动态绑定

不能直接把普通成员函数传递给 start_routine ，因为普通成员函数是 this call，而 start_routine 是普通的函数调用。

可以传递静态成员函数。
