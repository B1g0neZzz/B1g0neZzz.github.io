Timestamp

定义了时间戳的类

BOOST_STATIC_ASSERT

与普通 assert 不同之处在于， BOOST_STATIC_ASSERT 可以在编译时验证断言是否正确。

copyable

  • 是一个标记类
  • 以此为基类的子类可以拷贝，并且其本身必须是值类型
  • EBO(Empty Base Class Optimization) 空基类优化

此外，Timestamp 还继承了 boost:less_than_comparable<Timestamp> ，只要实现了 < 运算符，就相当于实现了 >、>=、<= 运算符

EBO

  • sizeof(Empty) == 1：编译器会给这个空类默默的安插一个 char，用来标识这个类。
  • sizeof(A) == 8：按字节对齐。
  • sizeof(B) == 4：空基类优化。

空基类只是不包含 non-static 成员变量，但是里面往往还包含有 enums、typedefs、static、non-virtual 函数。

NonCopyable

  • 禁用复制构造函数和赋值运算符，以此为父类的子类，因为父类对象不可拷贝和赋值，则本身也不能拷贝和赋值。

原子性操作

为什么需要原子性操作？

x++; 可以分为三个步骤：1. 从内存中读 x 的值到寄存器中；2. 对寄存器加 1；3. 把新值写回 x 所处的内存地址。

如下可能会导致 x 只增加了 1。

一个解决办法是锁，但是出现锁竞争会影响服务器性能。

可以使用原子操作。

gcc 原子性操作

volatile

当要求使用 volatile 声明的变量的值的时候，系统总是重新从它所在的内存读取数据，而不是使用保存在寄存器中的备份。 即使它前面的指令刚刚从该处去读过数据，而且读取的数据立刻被保存。

线程库

线程标识符

  • Linux 中，每个进程有一个 pid，类型 pid_t，由 getpid() 获得。Linux 下的 POSIX 线程也有一个 id，类型 pthread_t，由 pthread_self() 获得，该 id 由线程库维护，其 id 空间是各个进程独立的（即不同线程中的线程可能由相同的 id ）。Linux 中的 POSIX 线程库实现的线程其实也是一个进程（LWP），只是该进程与主进程（启动线程的进程）共享一些资源而已，比如代码段，数据段等。

  • 有时候我们可能需要知道线程的真实 pid。比如进程 P1 要向另外一个进程 P2 中的某个线程发送信号时，既不能使用 P2 的 pid，更不能使用线程的 pthread_id，而只能使用该进程的真实 pid，称为 tid。

  • 有一个函数 gettid() 可以得到 tid，但 glibc 并没有实现该函数，只能通过 Linux 的系统调用 syscall 来获取。

    • return syscal(SYS_gettid)

Thread 类图

  • started： 线程是否被启动
  • pthreadId： 线程 id
  • tid_： 线程 真实 id
  • func_： 线程的执行函数
  • name_： 线程的名称
  • numCreated： 已经创建的线程的数量（ 静态成员 ）
